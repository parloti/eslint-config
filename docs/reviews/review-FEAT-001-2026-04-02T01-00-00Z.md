## ⚠️ Issues

- [ ] The remediation is not internally complete: the architecture regression spec still asserts the deprecated rule alias instead of the new rule key. [src/infrastructure/architecture/index.spec.ts](../src/infrastructure/architecture/index.spec.ts#L17) still checks for boundaries/element-types even though the implementation note says the refactor switched to boundaries/dependencies in [docs/implementation/impl-FEAT-001-2026-04-02T00-30-00Z.md](../implementation/impl-FEAT-001-2026-04-02T00-30-00Z.md#L22) and the production code emits boundaries/dependencies in [src/infrastructure/architecture/boundaries.ts](../src/infrastructure/architecture/boundaries.ts#L67). That leaves the test suite validating the deprecated contract and weakens the claimed migration. Severity: Medium

- [ ] The new extension API is over-typed relative to its actual behavior, which makes it easy for consumers to pass valid-looking configuration that is silently ignored. [src/domain/types.ts](../src/domain/types.ts#L30) declares extend.elementTypes as a full RuleEntry, but [src/infrastructure/architecture/boundaries.ts](../src/infrastructure/architecture/boundaries.ts#L173) discards the extension severity and any non-rules option fields, only appending extensionOptions.rules at [src/infrastructure/architecture/boundaries.ts](../src/infrastructure/architecture/boundaries.ts#L179). A consumer can supply a different severity or default policy in extend.elementTypes and get no effect, with no validation or warning. That is a contract bug in a public API, not just a documentation gap. Severity: High

- Narrow BoundariesConfigExtension.elementTypes to the additive shape the code actually honors, or explicitly validate and reject unsupported severity/default overrides instead of silently dropping them.

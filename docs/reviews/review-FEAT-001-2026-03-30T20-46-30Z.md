## ⚠️ Issues

- [ ] The public package docs still describe the old consumer contract and actively instruct users to disable `jasmine` and `jest` via `disabledPlugins`, even though those plugins now default to disabled and the new `plugins` override object is the main addition. This makes the shipped API discoverability incomplete and can push consumers toward stale configuration patterns. Severity: High. References: [README.md](README.md#L36), [README.md](README.md#L41), [README.md](README.md#L140), [src/types.ts](src/types.ts#L24), [src/index.ts](src/index.ts#L1)

- [ ] The runtime config path duplicates plugin-state resolution and warning formatting locally in [src/config-factory.ts](src/config-factory.ts#L15) instead of reusing the tested logic in [src/utilities.ts](src/utilities.ts#L150) and [src/diagnostics.ts](src/diagnostics.ts#L71). That creates two sources of truth for the same default-disabled list and warning contract, so a future change can easily update one path and leave the other stale while tests continue to exercise the utility helper rather than the actual composition path. Severity: Medium. References: [src/config-factory.ts](src/config-factory.ts#L15), [src/config-factory.ts](src/config-factory.ts#L72), [src/utilities.ts](src/utilities.ts#L13), [src/utilities.ts](src/utilities.ts#L173), [src/diagnostics.ts](src/diagnostics.ts#L71)

## 💡 Suggestions

- Update [README.md](README.md) to document `plugins?: PluginStateOverrides`, call out that `jest` and `jasmine` are opt-in by default, and clarify whether `disabledPlugins` is legacy compatibility or still first-class.
- Collapse the duplicate runtime logic so config composition calls one shared resolver and one shared warning reporter. If the lint rule forced the split, move the shared implementation behind a small dedicated module rather than copying behavior into [src/config-factory.ts](src/config-factory.ts).

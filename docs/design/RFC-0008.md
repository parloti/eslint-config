# RFC-0008: Release Readiness and Governance

**Status:** Draft

## Objective

Define the release criteria and ongoing governance for the first public publication of `@codeperfect/eslint-config`.

## Release Baseline

Before publishing, the documentation set is the source of truth for:

- package identity and scope
- dependency ownership
- module taxonomy
- consumer API
- repository-specific architecture input
- validation and release gates

No public behavior should be published unless it has a corresponding documented home in this RFC set.

## First-Release Constraints

This package is being released as a clean first publication. The release should assume:

- no historical transition guide is required
- no compatibility commitments exist beyond the documented API
- undocumented behavior is not part of the supported surface
- consumer expectations are set by the README, RFCs, and generated API docs

The goal is to publish a smaller, clearer, and explicitly governed package surface.

## Release Gates

The package is release-ready when all of the following are true:

1. package metadata is accurate for npm and GitHub publication
2. README installation and usage examples match the shipped API
3. RFCs describe the supported architecture and consumer contract
4. `npm run validate` succeeds consistently
5. `npm run docs` regenerates the published documentation tree without drift

## Governance After Publication

After publication, changes to the package should continue to follow these rules:

- document public behavior before release
- keep the public API constrained to `config()` unless a new API adds materially new capability
- prefer removing undocumented ambiguity over preserving accidental behavior
- keep release validation and generated docs aligned with the published package

## Decision Principle

The target state is a smaller, clearer, and explicitly governed linting platform.

When implementation details are ambiguous, the published documentation defines the supported package behavior.

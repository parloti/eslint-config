# RFC-0003: Module Taxonomy and Composition Order

**Status:** Draft

## Objective

Define the logical modules that make up the final ESLint configuration and the order in which they compose.

This RFC intentionally describes **module responsibilities**, not the consumer API. API shape is defined in RFC-0004.

## Module Taxonomy

### Core Modules

The following modules are baseline and are enabled unless explicitly disabled:

- `eslint`
- `resolver`
- `typescript`

Responsibilities:

- base JavaScript and TypeScript safety
- import resolution for TypeScript projects
- foundational lint behavior used by every other module

### Architecture Modules

- `import-x`
- `boundaries`

Responsibilities:

- import/export hygiene
- repository-defined dependency direction and layer constraints

`boundaries` is architecturally separate because it requires repository-supplied topology data.

### Documentation Modules

- `comments`
- `jsdoc`

Responsibilities:

- disciplined ESLint directive usage
- API and example documentation consistency
- package-level JSDoc defaults and overrides

### Testing Modules

- `jasmine`
- `jest`
- `playwright`
- `vitest`

Responsibilities:

- test-framework-specific lint behavior
- file-scoped testing rules
- package-level testing defaults and overrides

These modules must target test files only.

### Domain Modules

- `rxjs-x`

Responsibilities:

- domain-specific reactive programming safety

### Style Modules

- `stylistic`
- `perfectionist`
- `unicorn`
- `prettier`

Responsibilities:

- style normalization
- deterministic ordering
- readability conventions
- Prettier interoperability

## Composition Order

The final configuration is composed in this order:

1. core modules
2. generic architecture modules (`import-x`)
3. documentation modules
4. testing modules
5. domain modules
6. style modules
7. repository-specific architecture overlays (`boundaries`)
8. consumer rule overrides

This ordering keeps foundational behavior early, scoping modules in the middle, and repository-specific or local overrides last.

## Deliberate Omissions

The platform does **not** expose separate `strict`, `all`, or other preset aliases at this stage.

Those aliases add naming surface without adding capability because the package already supports explicit module disablement and rule overrides through a single factory.

## Constraints

- Each module must have one clear responsibility.
- A module must not redefine the public API contract.
- Rule override precedence is always controlled by the final composition step.

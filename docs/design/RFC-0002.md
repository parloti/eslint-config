# RFC-0002: Dependency Ownership and Runtime Model

**Status:** Draft

## Objective

Define which dependencies are required from consumers, which integrations are owned by the package, and how the runtime behaves when a capability is unavailable.

## Dependency Classes

### Required Peer Dependencies

Every consuming repository must install the baseline toolchain required for flat-config execution:

- `eslint`
- `@eslint/js`
- `typescript`
- `typescript-eslint`
- `jiti` when the repository evaluates `eslint.config.ts`

These are required because they define the core execution environment for the config package.

### Package-Owned Optional Integrations

`@codeperfect/eslint-config` owns the lifecycle of optional integrations used by specific modules, including:

- `eslint-import-resolver-typescript`
- `eslint-plugin-boundaries`
- `eslint-plugin-import-x`
- `@eslint-community/eslint-plugin-eslint-comments`
- `eslint-plugin-jsdoc`
- `eslint-plugin-jasmine`
- `eslint-plugin-jest`
- `eslint-plugin-playwright`
- `@vitest/eslint-plugin`
- `eslint-plugin-rxjs-x`
- `@stylistic/eslint-plugin`
- `eslint-plugin-perfectionist`
- `eslint-plugin-unicorn`
- `eslint-plugin-prettier`

Consumers should not need to curate versions for these integrations individually.

### Repository-Owned Inputs

Some behavior cannot be packaged as a dependency because it is repository-specific. Those inputs remain consumer-owned, for example:

- `boundaries` element definitions
- repository file globs and ignore patterns
- rule overrides and local module disablement

## Runtime Rules

### Missing Required Peers

If a required peer dependency is unavailable, configuration creation must fail immediately. The package cannot safely continue when the baseline ESLint or TypeScript runtime is incomplete.

### Missing Optional Integrations

If an optional integration is unavailable, the affected module must resolve to `[]` and leave the rest of the config intact.

The package must not crash the entire configuration for an optional capability that a repository is not using. This behavior is especially important for test-framework-specific modules and other environment-dependent integrations.

### Missing Repository Input

If a module requires repository input and that input is not provided, the package must skip only that module and emit a clear diagnostic explaining how to configure or disable it.

`boundaries` is the primary example of this behavior.

## Rationale

This model keeps the package authoritative without forcing every repository to solve dependency management or architecture modeling from scratch.

It also separates three distinct concerns that were previously conflated:

- baseline runtime availability
- optional integration availability
- repository-specific configuration completeness

## Installation Example

Consumer repositories install the core toolchain and the config package:

```bash
npm install -D @codeperfect/eslint-config @eslint/js eslint jiti typescript typescript-eslint
```

No additional plugin installation should be required for the documented package-owned integrations.

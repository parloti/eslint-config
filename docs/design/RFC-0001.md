# RFC-0001: @codeperfect/eslint-config Package Charter

**Status:** Draft  
**Author:** Alex Parloti  
**Target:** ESLint Flat Config (ESLint ≥ 9)  
**Audience:** Platform Engineering, Application Engineers

## Executive Summary

This RFC defines the product boundary for `@codeperfect/eslint-config`.

The package is a **shareable flat-config package** for TypeScript repositories. It composes curated third-party ESLint integrations behind a single configuration factory for consuming repositories.

This replaces any plugin-centric framing. The package is **not** a generic `eslint-plugin-<org>` wrapper and does not ship package-owned ESLint rules as part of its public surface.

## Problem Statement

Repositories currently re-create or partially copy lint configuration, plugin wiring, and architectural rules. That creates:

- configuration drift
- inconsistent rule coverage
- uneven upgrade cadence
- duplicated maintenance for shared policy
- unclear ownership when local overrides diverge from platform defaults

## Goals

1. Provide one documented source of truth for lint policy across TypeScript repositories.
2. Centralize maintenance of third-party ESLint integrations and shared config composition.
3. Allow repository-level opt-out for modules that do not apply to a given codebase.
4. Support repository-specific architecture inputs without forcing one folder topology on every consumer.
5. Keep the public API small, typed, and stable.

## Non-Goals

The platform will not:

- support legacy `.eslintrc` formats
- replace ESLint core or `typescript-eslint`
- fork third-party plugins unless upstream becomes non-viable
- impose one universal architecture model on every repository
- expose multiple overlapping consumer APIs for the same outcome

## System Boundary

### Platform Responsibilities

The package owns:

- composition of flat-config modules into one final ESLint config
- integration of supported third-party plugins
- versioning, testing, and release validation

### Repository Responsibilities

A consuming repository owns:

- installation of required peer dependencies
- repository-specific architecture input when enabling `boundaries`
- local selection of disabled modules
- explicit rule overrides when a repository needs stricter or looser behavior

## Design Principles

1. **Config-first architecture**: configuration composition is the primary product surface.
2. **Typed extensibility**: repository inputs must be type-safe and intentionally narrow.
3. **Deterministic composition**: the same inputs must produce the same config ordering and rule set.
4. **Minimal public surface**: one factory is preferred over several aliases or preset wrappers.
5. **Documentation-first governance**: architectural behavior must be described in RFCs before it becomes package policy.

## Deliverable

A versioned npm package:

`@codeperfect/eslint-config`

used as the standard linting platform for TypeScript repositories.

## Success Criteria

The package is successful when:

- repositories no longer maintain duplicated ESLint integration code
- module enablement and override behavior are documented and testable
- package decisions are driven by RFCs instead of ad hoc local config changes

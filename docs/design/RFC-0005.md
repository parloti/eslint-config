# RFC-0005: Repository Architecture Input for `boundaries`

**Status:** Draft

## Objective

Define how repositories provide architecture topology to the `boundaries` module.

## Problem

`boundaries/elements` and `boundaries/element-types` are only meaningful when they describe the actual structure of a repository.

A generic default layer graph creates false confidence: it appears to enforce architecture while silently modeling the wrong one.

## Decision

The package will **not** ship a default repository architecture model.

Instead:

- `import-x` provides generic import hygiene
- `boundaries` activates only when the repository supplies explicit topology data

## Required Input Shape

Repositories enabling `boundaries` must provide:

- `files`: the file set the rules apply to
- `elements`: the repository's element descriptors
- `elementTypes`: the allowed dependency directions

Repositories may also provide:

- `ignores`: files or paths excluded from boundaries evaluation

## Example

```ts
import { config } from "@codeperfect/eslint-config";

export default config({
  boundaries: {
    files: ["src/**/*.ts"],
    elements: [
      { type: "shared", pattern: "src/shared/**" },
      { type: "domain", pattern: "src/domain/**" },
      { type: "app", pattern: "src/app/**" },
    ],
    elementTypes: [
      "error",
      {
        default: "disallow",
        rules: [
          { allow: ["shared"], from: ["domain", "app"] },
          { allow: ["domain"], from: ["app"] },
        ],
      },
    ],
  },
});
```

## Validation Rules

- Missing `elements` or `elementTypes` means `boundaries` is skipped.
- The package must emit a clear diagnostic telling the repository to either provide `boundaries` input or disable the module.
- Repository architecture input must remain a consumer concern; the package must not infer topology from folder names.

## Rationale

This approach preserves architectural correctness over convenience.

Adoption remains straightforward because repositories can begin with generic architecture modules and add `boundaries` once their topology is documented.
